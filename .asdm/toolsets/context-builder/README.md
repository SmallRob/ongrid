# ASDM Toolset - Context Builder (大型代码库版)

toolset-id: context-builder
toolset-name: Context Builder
version: 0.3.0
updated-date: 2026-04-14
toolset-description: 适配大型代码库（200万行+）的上下文构建工具，采用三明治架构和渐进式披露机制。

---

## 1. 概述

### 1.1 核心设计理念

**传统方案的痛点**：

| 问题 | 原因 |
|------|------|
| 全量生成内容不全 | 200万行代码一次性分析，AI注意力分散 |
| 拆分过细构建时间长 | 过度模块化导致重复分析和生成 |
| 使用时上下文过多 | 不区分任务类型，全部加载 |

**Context Builder v0.3 的解决方案**：

> 不是让 AI "知道一切"，而是建立一个**高效的导航系统**，让 AI 在需要时快速找到相关信息。

### 1.2 三明治架构

```
┌─────────────────────────────────────────────────────────────────┐
│                    三明治上下文架构                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  L1: 项目顶层索引（< 2KB，极轻量）                         │ │
│  │  - 项目一句话描述 + 技术栈                                 │ │
│  │  - 领域列表 + 入口链接                                    │ │
│  │  → AI 启动时必读，建立全局认知                            │ │
│  └───────────────────────────────────────────────────────────┘ │
│                              ↓                                  │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  L2: 领域索引（< 1KB/领域，按需加载）                     │ │
│  │  - 领域一句话描述                                         │ │
│  │  - 子模块列表 + 入口链接                                 │ │
│  │  → 任务涉及某领域时加载                                  │ │
│  └───────────────────────────────────────────────────────────┐ │
│                              ↓                                  │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  L3: 详细内容（< 5KB/文件，执行时按需加载）               │ │
│  │  - 实体定义、API详情、流程图                              │ │
│  │  → 遇到具体实现细节时加载                                │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 渐进式披露机制

```
用户启动 AI
     ↓
[读取 L1: 项目顶层索引]  →  建立全局认知，找到导航
     ↓
用户描述任务（如：修改用户模块的登录逻辑）
     ↓
[读取 L2: 用户域索引]  →  了解领域结构和子模块
     ↓
AI 执行任务，遇到具体实现
     ↓
[按需读取 L3: 相关详细内容]  →  获取具体实现细节
```

## 2. 核心原则

### 2.1 粒度控制黄金数字

| 指标 | 建议值 | 理由 |
|------|--------|------|
| L1 顶层索引 | < 2KB | 启动时必读，必须轻量 |
| L2 领域索引 | < 1KB/领域 | 按需加载，越轻越好 |
| L3 详细内容 | < 5KB/文件 | 执行时按需，高信息密度 |
| 单次加载总量 | < 50KB | 留空间给对话和源码 |
| 领域数量 | 5-10 个 | 符合认知负荷 |

### 2.2 按"决策域"划分，而非按"代码结构"

```
❌ 不合理的划分（按代码结构）
├── module-A/
│   ├── submodule-A1/
│   ├── submodule-A2/
│   └── ...
└── module-B/

✓ 合理的划分（按决策域）
├── 用户域/           # 同一业务决策相关
├── 订单域/
├── 支付域/
├── 基础设施/         # 技术基础设施独立
└── 跨域/             # 横切关注点
```

### 2.3 任务粒度与上下文粒度匹配

| 任务类型 | 需要加载的上下文 |
|----------|------------------|
| 理解项目全貌 | L1 顶层索引 |
| 规划新功能 | L1 + 相关领域 L2 |
| 修改单个组件 | L1 + L2 + 相关 L3 |
| 修改跨域逻辑 | L1 + 多个 L2 + 接口定义 |
| 重构 | 领域边界 + 依赖关系 + 策略 |

## 4. 工作流程

### 4.1 推荐工作流

```markdown
## 步骤 1: 初始化项目索引
/asdm-context-init
→ 生成 L1: 项目顶层索引（极轻量）

## 步骤 2: 构建领域索引（按需）
/asdm-domain-index-build <领域名>
→ 生成 L2: 领域索引（轻量）

## 步骤 3: 构建领域详细内容（按需）
/asdm-domain-context-build <领域名>
→ 生成 L3: 详细内容（按需加载）

## 步骤 4: 更新上下文
/asdm-context-update <变更类型>
→ 增量更新相关层级的上下文
```

### 4.2 命令参考

| 命令 | 描述 | 适用场景 |
|------|------|----------|
| `/asdm-context-init` | 初始化项目顶层索引 | 首次使用或项目结构大幅变化 |
| `/asdm-domain-index-build` | 构建单个领域索引 | 首次访问某领域或领域结构变化 |
| `/asdm-domain-context-build` | 构建领域详细内容 | 执行具体任务需要详细上下文 |
| `/asdm-context-update` | 更新变更的上下文 | 代码变更后的增量更新 |

## 5. 上下文文件结构

### 5.1 生成的上下文目录结构

```
.asdm/contexts/
├── index.md                    # L1: 项目顶层索引（极轻量）
│
├── domains/                    # L2: 领域索引目录
│   ├── user-domain.md         # 用户域索引
│   ├── order-domain.md        # 订单域索引
│   ├── payment-domain.md      # 支付域索引
│   └── infrastructure/         # 基础设施域
│       ├── cache-domain.md
│       ├── message-domain.md
│       └── ...
│
├── domain-details/            # L3: 详细内容目录
│   ├── user-domain/           # 用户域详细内容
│   │   ├── index.md          # 领域详情索引
│   │   ├── entities.md       # 实体定义
│   │   ├── services.md       # 服务接口
│   │   ├── apis.md           # API 定义
│   │   └── flows.md          # 业务流程
│   ├── order-domain/
│   │   └── ...
│   └── ...
│
└── workspace/                 # 工作区级别汇总
    ├── architecture.md
    ├── deployment.md
    └── standards/
        ├── coding-style.md
        └── project-structure.md
```

### 5.2 各层级文件大小控制

| 层级 | 文件 | 大小限制 | 内容要求 |
|------|------|----------|----------|
| L1 | `index.md` | < 2KB | 项目概览、技术栈、领域列表 |
| L2 | `domains/*.md` | < 1KB | 领域描述、子模块列表、入口链接 |
| L3 | `domain-details/*/*.md` | < 5KB | 具体定义、详细说明 |

## 6. 与 Project Context Analysis 的关系

Project Context Analysis 工具集是对 Context Builder 的增强，专注于**批量生成**和**状态追踪**。

| 工具集 | 定位 | 适用场景 |
|--------|------|----------|
| Context Builder | 核心上下文构建 | 按需生成、轻量级 |
| Project Context Analysis | 批量上下文构建 | 大型项目、需要状态管理 |

**推荐组合使用**：
1. 使用 Context Builder 定义上下文结构规范
2. 使用 Project Context Analysis 进行批量生成
