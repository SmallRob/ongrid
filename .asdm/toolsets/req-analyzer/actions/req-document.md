# req-document 操作指令

## 目的
本指令引导 AI 模型通过整合澄清、拆解和原型阶段的所有产出物来编制标准化的需求分析文档。它生成一个综合的 Markdown 文档，适用于利益相关者评审和下游开发团队。

## 语言检测

在生成任何内容之前，必须检测并使用当前环境的响应语言：

1. **检测响应语言**：分析环境设置，确定主要语言：
   - 检查系统/用户语言设置或环境配置
   - 识别项目文档和注释中使用的主要语言
   - 根据工作区上下文确定语言偏好

2. **保持语言一致性**：确保所有生成内容使用检测到的语言：
   - 所有 Markdown 文件、注释和文档均使用同一语言
   - 在所有生成文件中保持语言一致性
   - 遵循检测语言的书写规范和格式要求

3. **支持的语言**：
   - 中文（zh）
   - 英文（en）
   - 其他根据环境检测所需的语言

**重要**：语言检测是所有内容生成前的**第一步**。整个流程中的所有输出必须始终使用检测到的语言。

## 上下文注入

在生成需求文档之前，AI 模型应读取所有前一阶段的产出物。这确保文档是完整分析过程的综合汇编。

### 需读取的上下文文件（必需）

1. **澄清记录**（必需）
   - 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/clarification-notes.md`
   - 用途：提取业务目标、用户角色、功能范围、约束、假设条件

2. **特性列表**（必需）
   - 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/feature-list.md`
   - 用途：提取特性定义、优先级、复杂度、依赖关系

3. **系统功能清单**（可选 - 如已运行 `/req-system-mapping` 则读取）
   - 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/system-function-list.md`
   - 用途：提取各系统的职责边界、功能归属、集成方式，用于生成"系统集成说明"章节

4. **系统总览**（可选 - 如已运行 `/req-system-mapping` 则读取）
   - 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/system-overview.md`
   - 用途：提取系统清单和功能分布统计

5. **原型文件**（可选）
   - 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/prototypes/*.html`
   - 用途：参考原型页面结构进行用户交互说明

6. **渐进式上下文读取**（可选 - 按需）
   - `.asdm/contexts/index.md` — 与现有项目规范保持一致
   - `.asdm/contexts/architecture.md` — 参考技术架构约束

### 渐进式上下文加载策略

1. **初始阶段**：读取澄清记录和特性列表（均为必需）
2. **系统阶段**：检查 `system-function-list.md` 和 `system-overview.md` 是否存在，如存在则读取
3. **原型阶段**：扫描 prototypes 目录获取可用的 HTML 文件
4. **项目阶段**：读取项目上下文文件获取文档样式指导（如果有）

**重要**：生成文档前，澄清记录和特性列表必须都存在。如果任一缺失，告知用户需要完成的前置步骤。

## 模式选择

本工具支持两种模式,根据输入内容和前置条件自动选择:

### 模式 1: 标准流程（默认）

**触发条件**:
- 用户提供了需求 ID（例如：`/req-document REQ-001-在线预约挂号`）
- 且澄清记录和特性列表文件都已存在

**要求**:
- 必须先完成 `/req-clarify` 和 `/req-breakdown`
- 需要前置产物：`clarification-notes.md`、`feature-list.md`

**适合场景**:
- 经过完整澄清和拆解的复杂需求
- 需要严格的追溯性和完整性
- 需求文档用于正式评审和归档

### 模式 2: 快速模式（新增）

**触发条件**:
- 用户直接提供需求描述（例如：`/req-document 需求:实现用户登录功能,包括账号密码登录和手机验证码登录`）
- 或提供了需求 ID 但前置文件不存在,用户选择快速模式

**特点**:
- 无需前置产物,自动从需求描述生成最小化澄清记录和特性列表
- 生成简化版需求文档
- 快速产出文档,适合需求已明确的场景

**适合场景**:
- 需求已经明确,无需澄清
- 快速生成需求文档用于沟通确认
- 简单功能的需求记录

## 文档生成步骤

### 1. 接收输入并确定模式

从用户处获取输入，可以是以下两种形式之一：

#### 输入形式 A: 需求 ID（标准流程）

```
/req-document REQ-001-在线预约挂号
```

参数：
- **需求 ID**（必需）：例如 `REQ-001-在线预约挂号`

#### 输入形式 B: 需求描述（快速模式）

```
/req-document 需求:实现用户登录功能,包括账号密码登录和手机验证码登录
```

参数：
- **需求描述**（必需）：自然语言描述的需求内容
- 前缀关键词可以是：`需求:`、`需求描述:`、或直接描述
- 可选附加原型文件：`--prototype=path/to/prototype.html`

#### 输入形式 C: 未提供任何参数

列出可用需求：

```
当前已有的需求（来自 requirements-list.md）：
- REQ-001-<name> (CLARIFIED / BREAKDOWN / DOCUMENTED)
- REQ-002-<name> (CLARIFIED / BREAKDOWN)

请指定需求 ID，或提供需求描述进入快速模式。
```

### 2. 校验前置条件并确定模式

#### 情况 A: 用户提供了需求 ID

检查必需文件是否存在：
- 澄清记录：`.asdm/workspace/requirements/REQ-<id>-<name>/clarification-notes.md`
- 特性列表：`.asdm/workspace/requirements/REQ-<id>-<name>/feature-list.md`

**如果两个文件都存在** → 进入标准流程：
- 读取所有前一阶段产出物
- 继续步骤 3

**如果任一文件缺失** → 询问用户：

```
缺少前置文件：
- 澄清记录: <存在/不存在>
- 特性列表: <存在/不存在>

请选择操作模式：
1. 标准流程 - 先运行前置命令生成缺失文件（推荐用于复杂需求）
   - 缺少澄清记录 → 运行 /req-clarify
   - 缺少特性列表 → 运行 /req-breakdown
2. 快速模式 - 直接从需求描述生成简化文档（适合需求已明确的简单场景）

请输入 1 或 2，或提供需求描述。
```

- 如果用户选择 1 → 建议运行相应命令，停止流程
- 如果用户选择 2 或提供需求描述 → 进入快速模式

#### 情况 B: 用户提供了需求描述

直接进入快速模式：
- 跳过澄清记录和特性列表读取
- 自动生成需求 ID（格式：`REQ-QUICK-<timestamp>` 或用户自定义）
- 在步骤 3 中自动生成最小化前置文件

### 3. 加载所有产出物

#### 标准流程：读取前一阶段产物

读取并从所有可用产出物中提取信息：

##### 3.1 从澄清记录提取
- 业务目标和预期成果
- 用户角色和使用场景
- 功能需求和边界
- 非功能约束
- 假设条件列表
- 待确认事项 / 开放问题

##### 3.2 从特性列表提取
- 特性 ID、名称、描述
- 优先级和预估复杂度
- 用户角色分配
- 依赖关系

##### 3.3 从系统功能清单提取（如有）
- 涉及系统清单（系统名称、系统类型）
- 各系统的功能归属（自研/集成/委托）
- 跨系统的功能交互关系
- 系统功能分布统计

##### 3.4 从原型提取（如有）
- 页面结构说明
- 关键交互流程
- UI 组件布局

#### 快速模式：生成最小化前置文件

从需求描述中自动生成最小化的澄清记录和特性列表：

##### 步骤 3.1 生成最小化澄清记录

## 执行指南

### 何时使用本操作

在以下情况下使用本操作：
- 澄清和拆解阶段已完成
- 需要综合需求文档供利益相关者评审
- 为开发团队准备交接材料
- 归档需求分析过程

### 文档质量指南

1. **完整性**：文档应覆盖前一阶段的所有信息
2. **一致性**：术语和描述应在各章节间保持一致
3. **可追溯性**：每个需求应可追溯到其来源（澄清记录）
4. **可读性**：使用清晰的语言、表格和结构化格式
5. **可操作性**：文档应为下游团队提供足够的细节

### 文档定制

- 如果**未**生成原型：跳过第 8 节（原型参考）
- 如果**未**运行 `/req-system-mapping`：跳过第 6 节（系统集成说明）
- 如果**已**运行 `/req-system-mapping`：第 6 节必须填写，不得省略
- 如果澄清记录包含大量问答：在附录中包含完整记录
- 如果需求较简单：简化第 5-10 节（附录保留）

## 使用方式

使用本指令时，AI 模型应：

### 标准流程

1. 检测响应语言
2. 从用户处获取需求 ID
3. 校验前置条件（澄清记录和特性列表存在）
4. 加载前一阶段的所有产出物
5. 如有项目上下文则读取（渐进式加载）
6. 按模板结构生成需求文档
7. 保存文档
8. 更新需求列表
9. 向用户展示摘要

### 快速模式流程

1. 检测响应语言
2. 从用户处获取需求描述（或检测到缺少前置文件后进入）
3. 自动生成最小化澄清记录和特性列表
4. 保存临时前置文件
5. 从临时文件中提取信息
6. 生成简化版需求文档（跳过非必需章节）
7. 保存文档
8. 创建需求记录到 requirements-list.md
9. 向用户展示摘要

## 输出摘要

完成后，将生成以下产出物：
- 需求文档：`.asdm/workspace/requirements/REQ-<id>-<name>/requirement-doc.md`
- 已更新的需求列表：`.asdm/workspace/requirements/requirements-list.md`

所有文件均保存在 `.asdm/workspace/requirements/` 目录下。
