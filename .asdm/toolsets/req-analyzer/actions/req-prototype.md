# req-prototype 操作指令

## 目的
本指令引导 AI 模型根据特性列表为指定特性生成可交互的 HTML 原型。它生成可视化的、可点击的页面原型，帮助利益相关者通过交互原型验证需求理解的准确性。

## 语言检测

在生成任何内容之前，必须检测并使用当前环境的响应语言：

1. **检测响应语言**：分析环境设置，确定主要语言：
   - 检查系统/用户语言设置或环境配置
   - 识别项目文档和注释中使用的主要语言
   - 根据工作区上下文确定语言偏好

2. **保持语言一致性**：确保所有生成内容使用检测到的语言：
   - 所有 Markdown 文件、注释和文档均使用同一语言
   - 在所有生成文件中保持语言一致性
   - 遵循检测语言的书写规范和格式要求

3. **支持的语言**：
   - 中文（zh）
   - 英文（en）
   - 其他根据环境检测所需的语言

**重要**：语言检测是所有内容生成前的**第一步**。整个流程中的所有输出必须始终使用检测到的语言。

## 上下文注入

在生成原型之前，AI 模型应读取特性列表和澄清记录，以了解特性详情和用户交互需求。

### 需读取的上下文文件（必需）

1. **特性列表**（必需）
   - 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/feature-list.md`
   - 用途：加载特性定义、描述、用户角色和外部服务依赖

2. **澄清记录**（推荐）
   - 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/clarification-notes.md`
   - 用途：了解业务背景和用户场景

3. **样式规范**（必须）
   - 路径：`.asdm/toolsets/req-analyzer/spec/style-guide-spec.md`
   - 用途：确保生成的原型符合项目视觉规范（配色、字体、间距、Tailwind 工具类）

4. **项目组件参考**（推荐）
   - 路径：`components/` 目录
   - 用途：参考现有 React + Tailwind CSS 组件实现，保持风格一致

5. **渐进式上下文读取**（可选 - 按需）
   - `.asdm/contexts/index.md` — 与现有项目风格保持一致

### 渐进式上下文加载策略

1. **初始阶段**：读取特性列表（必需）以了解需要生成原型的内容
2. **详细阶段**：读取澄清记录获取额外上下文
3. **样式阶段**：读取样式规范获取配色、字体、间距等视觉规范和 Tailwind 工具类映射
4. **组件参考**：浏览 `components/` 目录参考现有组件实现，使用 Tailwind CSS + Framer Motion 构建

**重要**：开始生成原型前必须先读取特性列表。如果特性列表不存在，告知用户先运行 `/req-breakdown`。

## 模式选择

本工具支持两种模式,根据输入内容和前置条件自动选择:

### 模式 1: 标准流程（默认）

**触发条件**:
- 用户提供了需求 ID（例如：`/req-prototype REQ-001-在线预约挂号`）
- 且特性列表文件已存在

**要求**:
- 必须先完成 `/req-clarify` 和 `/req-breakdown`
- 需要前置产物：`feature-list.md`、`clarification-notes.md`

**适合场景**:
- 经过完整澄清和拆解的复杂需求
- 需要严格的追溯性和完整性

### 模式 2: 快速模式（新增）

**触发条件**:
- 用户直接提供需求描述（例如：`/req-prototype 需求:实现用户登录功能,包括账号密码登录和手机验证码登录`）
- 或提供了需求 ID 但特性列表不存在,用户选择快速模式

**特点**:
- 无需前置产物,自动从需求描述中提取特性
- 在内存中生成临时特性列表
- 快速产出原型,适合需求已明确的场景

**适合场景**:
- 需求已经明确,无需澄清
- 快速验证需求理解
- 简单功能的原型设计

## 原型生成步骤

### 1. 接收输入并确定模式

从用户处获取输入，可以是以下两种形式之一：

#### 输入形式 A: 需求 ID（标准流程）

```
/req-prototype REQ-001-在线预约挂号 [--feature=F-001 | --all]
```

参数：
- **需求 ID**（必需）：例如 `REQ-001-在线预约挂号`
- **特性选择**（可选）：`--feature=F-001` 或 `--all`（默认生成所有特性）

#### 输入形式 B: 需求描述（快速模式）

```
/req-prototype 需求:实现用户登录功能,包括账号密码登录和手机验证码登录
```

参数：
- **需求描述**（必需）：自然语言描述的需求内容
- 前缀关键词可以是：`需求:`、`需求描述:`、或直接描述

#### 输入形式 C: 未提供任何参数

列出可用需求：

```
当前已有的需求（来自 requirements-list.md）：
- REQ-001-<name> (CLARIFIED / BREAKDOWN / PROTOTYPED)
- REQ-002-<name> (CLARIFIED / BREAKDOWN)

请指定需求 ID，或提供需求描述进入快速模式。
```

### 2. 校验前置条件并确定模式

#### 情况 A: 用户提供了需求 ID

检查特性列表文件是否存在：
- 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/feature-list.md`

**如果特性列表存在** → 进入标准流程：
- 读取特性列表和澄清记录（如有）
- 继续步骤 3

**如果特性列表不存在** → 询问用户：

```
未找到特性列表文件（feature-list.md）。

请选择操作模式：
1. 标准流程 - 先运行 /req-breakdown 生成特性列表（推荐用于复杂需求）
2. 快速模式 - 直接从需求描述生成原型（适合需求已明确的简单场景）

请输入 1 或 2，或提供需求描述。
```

- 如果用户选择 1 → 建议运行 `/req-breakdown`，停止流程
- 如果用户选择 2 或提供需求描述 → 进入快速模式

#### 情况 B: 用户提供了需求描述

直接进入快速模式：
- 跳过特性列表读取
- 自动生成需求 ID（格式：`REQ-QUICK-<timestamp>` 或用户自定义）
- 在步骤 3 中自动从需求描述提取特性

### 3. 加载特性定义

#### 标准流程：从特性列表加载

读取特性列表并提取每个要生成原型的特性：
- 特性名称和描述
- 与该特性交互的用户角色
- 对其他特性的依赖
- 预期的用户工作流
- **外部服务依赖**（Gemini AI/Capacitor/Three.js）

#### 快速模式：自动提取特性

从需求描述中自动提取特性：

**步骤 3.1 分析需求描述**

识别需求描述中的关键信息：
- **功能点**：识别主要功能和子功能
- **用户角色**：识别涉及的用户类型
- **交互场景**：识别用户操作流程
- **数据实体**：识别涉及的数据对象

**步骤 3.2 生成临时特性列表**

在内存中构建特性列表（结构同标准特性列表）：

```markdown
# 特性列表

**需求编号**: REQ-QUICK-<timestamp>（或用户指定的 ID）
**需求名称**: <从描述提取>
**生成时间**: YYYY-MM-DD HH:mm:ss
**模式**: 快速模式

## 特性清单

| 特性ID | 特性名称 | 优先级 | 预估复杂度 | 描述 |
|--------|----------|--------|-----------|------|
| F-001 | <特性1> | HIGH | MEDIUM | <描述> |
| F-002 | <特性2> | MEDIUM | SMALL | <描述> |

## 特性详情

### F-001: <特性名称>

**优先级**: HIGH
**预估复杂度**: MEDIUM
**用户角色**: <角色>

**功能描述**:
<详细描述>

**功能要点**:
- <要点1>
- <要点2>

**依赖特性**: 无
```

**步骤 3.3 保存临时特性列表**

将临时特性列表保存到文件：
- 路径：`.asdm/workspace/requirements/REQ-QUICK-<timestamp>/feature-list.md`
- 标注为"快速模式生成"，供后续 `/req-document` 使用

**重要**：
- 快速模式生成的特性列表**不经过澄清**,仅基于需求描述自动提取
- 如果后续需要补充澄清，可运行 `/req-clarify` 后再运行 `/req-breakdown` 覆盖
- 临时特性列表仍会保存到文件，便于追溯和后续补充

### 4. 分析代码库结构（可选）

如果项目已有代码库，分析代码库以生成符合项目风格的原型：

#### 4.1 分析项目技术栈

识别项目使用的技术：
- 前端框架（React、Vue、Angular 等）
- 样式方案（Tailwind CSS v4、CSS、Sass 等）
- 动画方案（Framer Motion 等）
- 构建工具（Webpack、Vite 等）

#### 4.2 分析现有组件

分析代码库中的现有组件：
- 组件目录结构
- 可复用的 UI 组件
- 布局模式
- 样式规范

> **📦 组件库参考**：如果需要了解具体 UI 组件的实现细节，参考 `components/` 目录下的现有组件。项目使用 React 19 + Tailwind CSS v4 + Framer Motion，不依赖第三方 UI 组件库。

#### 4.3 分析页面结构

分析现有页面的结构模式：
- 页面布局模式
- 导航结构
- 表单和表格样式
- 交互模式

**重要**：如果代码库分析失败或无法读取，使用默认的简洁现代设计风格。

### 5. 设计页面结构

为每个特性设计页面结构：

#### 5.1 确定原型文件结构

根据项目复杂度选择原型文件结构：

**选项 1：单文件结构（默认）**
- 所有 HTML、CSS、JavaScript 内嵌到一个 HTML 文件
- 适合简单原型和快速分享
- 文件路径：`.asdm/workspace/requirements/REQ-<id>-<name>/prototypes/<feature-id>-prototype.html`

**选项 2：多文件结构（可选）**
- HTML、CSS、JavaScript 分离到不同文件
- 适合复杂原型和后续迭代
- 目录结构：
  ```
  .asdm/workspace/requirements/REQ-<id>-<name>/prototypes/<feature-id>/
  ├── index.html
  ├── styles.css
  ├── scripts.js
  └── components/
      ├── header.html
      ├── footer.html
      └── ...
  ```

如果用户未指定，默认使用单文件结构。

#### 5.2 页面类型
根据特性性质，确定适当的页面类型：
- **列表页**：适用于涉及数据浏览的特性（如预约列表、医生列表）
- **表单页**：适用于涉及数据输入的特性（如注册表单、预约预订）
- **详情页**：适用于涉及数据查看的特性（如预约详情、用户资料）
- **仪表盘页**：适用于涉及数据概览的特性（如管理后台、统计信息）

#### 5.3 交互设计
为每个页面设计：
- 页面间的导航流程
- 关键 UI 元素（按钮、表单、表格、卡片）
- 用户交互模式（点击、提交、搜索、筛选）
- 反馈机制（成功消息、错误状态、加载状态）

#### 5.4 多系统交互处理

如果读取了系统功能清单，为每个涉及外部系统的操作设计：

- **数据标注**：在界面上标注外部系统数据来源（如"医生数据来自医院系统"、"支付数据来自支付平台"）
- **交互状态**：设计加载状态、超时提示、错误提示等交互反馈
- **模拟数据**：使用模拟数据展示外部系统返回结果
- **边界提示**：标注"待对接"、"Mock 数据"、"实际需调用 XX 系统"等标记
- **系统归属说明**：在原型底部或侧边栏添加说明框，列出当前页面涉及的系统及其职责

### 6. 生成 HTML 原型

为每个选中的特性生成可交互的 HTML 原型：

#### 6.1 HTML 结构

HTML 原型应包括：
- 完整的 HTML5 文档结构
- 内嵌 CSS 样式（现代、简洁设计）
- 内嵌 JavaScript 实现交互行为
- 响应式布局适配不同屏幕尺寸
- 中文标签和占位符文本

#### 6.2 样式指南

**如果样式规范文件存在（.asdm/toolsets/req-analyzer/spec/style-guide-spec.md），严格遵循规范中的定义**：

- **配色方案**：使用规范中定义的主色、文字色、功能色、中性色
- **字体规范**：使用规范的字体族、字号、字重、行高
- **间距规范**：使用规范定义的间距单位（4px/8px/16px/24px/32px）
- **圆角规范**：使用规范定义的圆角值（按钮 4px、卡片 8px、标签 12px）
- **按钮样式**：主要按钮、次要按钮、文字按钮的样式和交互状态
- **表单样式**：输入框、下拉框的边框、圆角、聚焦状态
- **表格样式**：表头、表格行、悬停状态的背景和文字颜色
- **标签样式**：成功、警告、错误、信息状态标签的配色
- **卡片样式**：边框、圆角、内边距、阴影

> **📦 组件库参考**：如需查看具体 UI 组件的实现代码，参考 `components/` 目录。项目使用 Tailwind CSS v4 工具类构建 UI，样式规范见 `style-guide-spec.md`。

**如果样式规范文件不存在**：
- 使用简洁、现代的设计风格
- 包含一致的颜色方案（主色、辅色、强调色）
- 使用适当的间距和排版
- 为交互元素包含 hover/focus 状态
- 支持基本响应式设计

#### 6.3 交互元素
- 导航链接应可点击（页面切换）
- 表单输入应有占位符文本
- 按钮应有点击反馈
- 标签/手风琴应可折叠
- 适当包含模拟数据（中文姓名、日期等）

#### 6.4 原型内容
- 使用真实的中文占位符内容，不要用 lorem ipsum
- 在表格和列表中包含示例数据
- 在相关位置同时展示空状态和填充状态
- 包含导航面包屑或侧边栏

#### 6.5 系统边界标注

如果涉及多系统交互，在原型中标注系统边界：

- **数据标注**：在显示外部数据的位置添加提示
  ```html
  <div class="external-data-badge">
    <span>🔗 数据来源: 医院信息系统 (HIS)</span>
  </div>
  ```

- **系统说明框**：在页面底部或侧边栏添加系统说明
  ```html
  <div class="system-info">
    <h3>📌 本页面涉及的系统</h3>
    <ul>
      <li><strong>本系统 (SYS-001)</strong>: 用户界面、预约流程</li>
      <li><strong>医院系统 (SYS-002)</strong>: 医生数据、排班信息 [待对接]</li>
      <li><strong>支付系统 (SYS-003)</strong>: 在线支付 [待对接]</li>
    </ul>
  </div>
  ```

- **外部操作标识**：为调用外部系统的操作添加特殊样式
  ```html
  <button class="external-call">
    💳 在线支付
    <small>→ 调用支付系统 API</small>
  </button>
  ```

- **加载状态提示**：为外部数据加载设计占位符
  ```html
  <div class="loading-placeholder">
    <span class="spinner"></span>
    <span>正在从医院系统加载医生数据...</span>
  </div>
  ```

### 7. 保存原型文件

#### 7.1 单文件结构（默认）

将每个原型保存至：
- 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/prototypes/<feature-id>-prototype.html`
- 格式：独立的 HTML 文件（CSS 和 JS 内嵌）

#### 7.2 多文件结构（可选）

如果选择多文件结构：
- 创建目录：`.asdm/workspace/requirements/REQ-<id>-<name>/prototypes/<feature-id>/`
- 保存主页面：`index.html`
- 保存样式文件：`styles.css`
- 保存脚本文件：`scripts.js`
- 保存组件文件：`components/<component-name>.html`

如不存在则创建目录：
```
mkdir -p .asdm/workspace/requirements/REQ-<id>-<name>/prototypes/
```

### 8. 生成页面结构文档

为每个原型生成简要的页面结构说明：
- 页面用途
- 关键 UI 组件
- 用户交互流程
- 关联页面

### 9. 启动预览服务器（可选）

如果生成了原型文件，可以启动本地预览服务器：

#### 9.1 检测可用端口

检测可用的本地端口：
- 默认使用 3000 端口
- 如果被占用，尝试下一个可用端口（3001、3002 等）

#### 9.2 启动 HTTP 服务器

使用 Python 或 Node.js 启动简单的 HTTP 服务器：

```bash
# 单文件结构
cd .asdm/workspace/requirements/REQ-<id>-<name>/prototypes/
python -m http.server <port>

# 多文件结构
cd .asdm/workspace/requirements/REQ-<id>-<name>/prototypes/<feature-id>/
python -m http.server <port>

# 或使用 Node.js (如果安装了 npx)
npx serve -p <port>
```

#### 9.3 在浏览器中打开预览

自动打开浏览器访问原型：
- URL：`http://localhost:<port>`（单文件）或 `http://localhost:<port>/<feature-id>/`（多文件）
- 使用系统默认浏览器

**重要**：如果启动服务器失败，直接提供文件路径供用户手动打开。

### 11. 验证清单

**⚠️ 在展示摘要前，必须执行验证清单**

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 语言一致性 | □ | 所有原型内容使用检测到的语言 |
| 特性覆盖完整 | □ | 所选的每个特性都已生成原型文件 |
| 核心交互覆盖 | □ | 每个原型覆盖了该特性的所有主要用户操作路径 |
| 用户角色覆盖 | □ | 原型中体现了特性所涉及的所有用户角色的视图 |
| 文件可访问 | □ | 所有 HTML 文件已保存，路径正确，可在浏览器打开 |
| 内容真实性 | □ | 使用真实中文占位内容，无 lorem ipsum |
| 交互可用 | □ | 关键按钮、导航、表单提交有可见的交互反馈 |
| 空状态处理 | □ | 列表/表格类页面包含空状态展示 |
| 样式规范遵循 | □ | 如果样式规范存在，严格遵循配色、字体、间距等规范 |

## 执行指南

### 何时使用本操作

在以下情况下使用本操作：
- 特性拆解已完成
- 需要可视化原型与利益相关者验证需求理解
- 希望在开发前探索用户交互模式
- 准备需求评审会议材料

### 原型质量指南

1. **真实性**：使用真实的中文内容和数据，不要用 lorem ipsum
2. **完整性**：覆盖该特性的所有关键用户交互场景
3. **一致性**：所有原型保持一致的样式
4. **可交互性**：关键交互应可使用（导航、表单提交反馈）
5. **可读性**：布局应清晰易懂

### 原型范围指南

- **HIGH 优先级**特性：生成完整的多页面原型
- **MEDIUM 优先级**特性：生成单页或两页原型
- **LOW 优先级**特性：生成简单的线框风格原型
- 聚焦用户-facing 功能，除非明确要求否则跳过后台/管理功能

## 使用方式

使用本指令时，AI 模型应：

### 标准流程

1. 检测响应语言
2. 从用户处获取需求 ID、特性选择和系统过滤选项（可选）
3. 校验前置条件（特性列表存在；如使用 --system 则需要系统功能清单）
4. 加载特性定义和系统映射信息（如有）
5. 分析代码库结构（可选，如果存在）
6. 为每个特性设计页面结构和多系统交互处理
7. 生成可交互 HTML 原型（包含系统边界标注）
8. 保存原型文件（单文件或多文件结构）
9. 生成页面结构文档
10. 启动预览服务器（可选）
11. **执行验证清单**（检查功能覆盖完整度和系统边界标注）
12. 向用户展示摘要

### 快速模式流程

1. 检测响应语言
2. 从用户处获取需求描述（或检测到缺少特性列表后进入）
3. 自动从需求描述提取特性，生成临时特性列表
4. 保存临时特性列表到文件
5. 分析代码库结构（可选，如果存在）
6. 为每个特性设计页面结构
7. 生成可交互 HTML 原型
8. 保存原型文件
9. 生成页面结构文档
10. 启动预览服务器（可选）
11. **执行验证清单**
12. 向用户展示摘要
