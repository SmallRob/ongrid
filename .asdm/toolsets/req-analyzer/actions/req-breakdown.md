# req-breakdown 操作指令

## 目的
本指令引导 AI 模型将已澄清的业务需求拆解为**可独立上线的业务功能单元**。

**核心理念**：
- 特性 = 可独立交付、独立测试、独立上线的完整业务功能
- 不是技术实现细节,不是开发任务分解
- 从用户价值角度,而非技术实现角度思考
- 每个特性都应该能产生可见的业务价值

## 语言检测

在生成任何内容之前，必须检测并使用当前环境的响应语言：

1. **检测响应语言**：分析环境设置，确定主要语言：
   - 检查系统/用户语言设置或环境配置
   - 识别项目文档和注释中使用的主要语言
   - 根据工作区上下文确定语言偏好

2. **保持语言一致性**：确保所有生成内容使用检测到的语言：
   - 所有 Markdown 文件、注释和文档均使用同一语言
   - 在所有生成文件中保持语言一致性
   - 遵循检测语言的书写规范和格式要求

3. **支持的语言**：
   - 英文（en）
   - 中文（zh）
   - 其他根据环境检测所需的语言

**重要**：语言检测是所有内容生成前的**第一步**。整个流程中的所有输出必须始终使用检测到的语言。

## 上下文注入

在执行特性拆解之前，AI 模型应读取澄清说明和项目上下文，以确保特性定义基于已澄清的需求，并与项目现有架构保持一致。

### 需读取的上下文文件（必需）

1. **澄清说明**（必需）
   - 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/clarification-notes.md`
   - 用途：加载已澄清的业务需求、假设条件和待定事项

2. **index.md**（推荐）
   - 路径：`.asdm/contexts/index.md`
   - 用途：了解项目概览和现有功能

3. **渐进式上下文读取**（可选 - 按需）
   - `.asdm/contexts/architecture.md` — 使特性与技术架构保持一致
   - `.asdm/contexts/data-models.md` — 识别数据依赖关系
   - `.asdm/contexts/api.md` — 识别集成点

### 渐进式上下文加载策略

1. **初始阶段**：读取澄清说明（必需）
2. **分析阶段**：读取 `index.md` 了解现有项目上下文
3. **依赖阶段**：如特性涉及特定技术领域，读取对应的上下文文件

**重要**：开始拆解之前必须先读取澄清说明。如果澄清说明不存在，请告知用户先运行 `/req-clarify`。

## 特性拆解步骤

### 1. 接收需求 ID

从用户处获取需求 ID：
- **必需**：需求 ID（例如：`REQ-001-在线预约挂号`）
- 格式：`REQ-<NNN>-<名称>`

如未提供，列出已有需求：

```
当前已有的需求（来自 requirements-list.md）：
- REQ-001-<name> (CLARIFIED)
- REQ-002-<name> (CLARIFIED)

请指定要进行特性拆解的需求 ID。
```

### 2. 校验澄清说明是否存在

检查澄清说明文件是否存在：
- 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/clarification-notes.md`

如文件不存在：
- 告知用户必须先完成澄清
- 建议运行 `/req-clarify` 生成澄清说明
- 终止拆解流程

### 2.5 检测增量更新场景

检查是否存在已有的 `feature-list.md`：
- 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/feature-list.md`

**如果文件已存在**（说明本次可能是重新拆解），向用户说明：

```
检测到该需求已有特性列表（feature-list.md）。

可能的场景：
1. 澄清说明已更新 → 需要重新拆解，更新特性列表
2. 用户主动触发重新拆解 → 确认后覆盖现有特性列表

是否继续并覆盖现有特性列表？（请确认）
```

用户确认后：
- 在新的 `feature-list.md` 顶部记录变更说明（版本号+1，注明"基于更新后的澄清说明重新拆解"）
- 继续执行后续步骤

**如果文件不存在**，直接继续执行步骤 3。

### 3. 加载并分析澄清说明

读取澄清说明，提取以下内容：

- 业务目标和预期成果
- 用户角色和使用场景
- 已识别的功能需求
- 非功能性约束
- 假设条件和待定事项

### 4. 识别特性

根据已澄清的需求,识别**可独立上线的业务功能单元**：

#### 4.1 特性识别标准（核心原则）

满足以下条件时才应识别为独立特性：

1. **独立可交付**
   - 可以独立开发、测试、部署
   - 上线后能产生独立的业务价值
   - 不依赖其他特性的完成才能交付价值

2. **用户可见的价值**
   - 用户能直接感知到这个功能的存在
   - 能用业务语言描述"这个功能让用户可以做什么"
   - 而不是"这个功能让系统可以怎样实现"

3. **业务完整性**
   - 包含了完成该功能所需的所有环节
   - 从用户视角看是一个完整的业务场景
   - 不会被拆解成碎片化的技术组件

4. **合理的粒度**
   - 既不能太粗（包含多个独立的业务价值点）
   - 也不能太细（变成开发任务或技术实现细节）

#### 4.2 特性识别的反模式（避免）

**不要拆分成以下内容**：

1. **技术组件**
   - ❌ "数据库表设计"
   - ❌ "后端接口开发"
   - ❌ "前端页面实现"
   - ✅ "用户注册功能"（包含前后端、数据库等所有技术实现）

2. **开发任务**
   - ❌ "实现关键字提取算法"
   - ❌ "开发查询接口"
   - ❌ "设计展示界面"
   - ✅ "审批历史意见引用"（完整的功能体验）

3. **数据流中的单一环节**
   - ❌ "数据同步接口"
   - ❌ "消息队列监听器"
   - ❌ "日志记录功能"
   - ✅ "跨系统数据同步"（完整的数据流转能力）

4. **辅助性功能独立拆分**
   - ❌ 将"审计日志"、"数据统计"作为独立特性
   - ✅ 将这些作为主要功能的一部分,或只在必要时作为独立特性（如审计是合规强需求时）

#### 4.3 特性命名规范

使用格式：`<动词+名词>`（例如："用户登录"、"在线预约"、"数据导出"）

**命名原则**：
- 从用户视角命名,体现用户能做什么
- 避免技术术语
- 体现业务价值

#### 4.3 特性 ID 规范

使用格式：`F-<NNN>`，其中 NNN 为需求内的顺序编号（001、002 等）

### 5. 校验特性数量和粒度

识别特性后，进行双重校验：

#### 5.1 数量校验
- 特性数量为 **3-8 个**：✅ 合适
- 特性数量为 1-2 个：⚠️ 可能过粗,检查是否可以进一步拆分
- 特性数量为 **9-10 个**：⚠️ 可接受,但建议合并或拆分需求
- 特性数量**超过 10 个**：❌ 需要调整
  - **停止**拆解流程
  - 向用户说明特性拆分过细或需求规模过大
  - 请用户调整拆分粒度或将需求拆分为更小的子需求

#### 5.2 粒度校验清单

使用以下问题自检每个特性的粒度是否合适：

1. **用户价值测试**
   - "上线这个特性后,用户能立即感知到什么价值?"
   - 如果答案是"用户感知不到",则粒度可能太细

2. **独立交付测试**
   - "这个特性可以单独上线吗?"
   - 如果答案是"不能,需要等其他特性一起",则需要合并或调整依赖

3. **业务完整性测试**
   - "这个功能从用户发起操作到看到结果,整个流程完整吗?"
   - 如果答案是"不完整,只做了其中一部分",则需要扩展或合并

4. **描述测试**
   - 能否用"用户可以XXX"的句式完整描述这个特性?
   - 如果描述变成"系统需要YYY"或"开发要做ZZZ",说明视角错了

#### 5.3 合并与拆分指南

**需要合并的情况**：
- A特性是B特性的前置条件,且两者单独交付都无意义
- 多个小特性共同完成一个用户场景
- 技术实现上紧密相关,必须一起交付

**需要拆分的情况**：
- 一个特性包含多个独立的用户价值点
- 不同用户角色对该特性的需求差异很大
- 不同场景下使用该特性的流程完全不同

### 6. 识别关联系统

为每个特性识别涉及的关联系统，作为后续 System Mapping 的基础输入。

#### 6.1 依赖系统清单

| 系统类型 | 说明 | 示例 |
|----------|------|------|
| **A+ 类** | 本需求的核心开发系统 | 需求管理系统、医疗问诊平台 |
| **A 类** | 与本需求有数据/流程交互的自研系统 | 工作流引擎、通知服务、用户中心 |
| **B 类** | 需对接的内部共享平台，非本团队负责 | 统一认证平台、数据中台、日志平台 |
| **C 类** | 通过 API/SDK 对接的第三方外部系统 | 微信支付、阿里云短信、HIS系统 |

#### 6.2 关联系统识别标准

满足以下任一条件时，应将系统列入关联系统：
- 特性需要调用该系统的接口或 API
- 特性产生的数据需要同步或共享到该系统
- 该系统的数据或状态变化会影响特性执行
- 特性需要在多个系统间协调流转（如流程审批）

#### 6.3 识别方法

1. **从澄清记录中提取**：查找"系统"、"平台"、"接口"、"对接"等关键词
2. **从用户角色推导**：如果某角色属于特定系统，该系统通常是关联系统
3. **从技术依赖识别**：消息队列、缓存、存储、外部 API 调用等都是关联点
4. **与项目上下文对照**：参考 `architecture.md` 中的系统架构图

#### 6.4 识别结果记录

将识别结果记录为以下格式：
- `<系统名称>`（系统类型）
- 示例：`需求管理系统（A+类）`、`工作流引擎（A类）`、`统一认证平台（B类）`、`微信支付（C类）`

### 6.5 识别外部服务依赖 (新增 - 针对 Nice Today 2.0 项目)

对每个特性,识别是否依赖外部服务或第三方技术:

#### 外部服务依赖类型

| 依赖类型 | 示例 | 关注点 |
|---------|------|--------|
| **AI 服务** | Gemini API (对话/分析/生成) | API调用频率、Token限制、响应格式、成本控制 |
| **设备 API** | Capacitor 插件(相机/定位/存储/通知) | 权限申请、平台兼容性(Android/Web)、降级方案 |
| **3D 渲染** | Three.js (虚拟展示/能量树) | 性能优化、FPS监控、内存管理、低端设备降级 |
| **第三方 SDK** | 支付/分享/统计/地图 | SDK版本、初始化配置、许可证合规 |
| **云服务** | 云存储/云函数/CDN | 网络延迟、离线支持、费用预估 |

#### 识别规则

**从特性描述中识别**：
- 提到「AI分析」「智能推荐」「对话」→ AI 服务依赖
- 提到「拍照」「定位」「本地存储」→ 设备 API 依赖
- 提到「3D展示」「虚拟场景」「粒子效果」→ 3D 渲染依赖
- 提到「支付」「分享」「地图导航」→ 第三方 SDK 依赖

**输出要求**：
在 feature-list.md 中为每个特性增加 `external_dependencies` 字段：
- 无依赖：`[]`
- 有依赖：`[{type: "AI服务", name: "Gemini API", purpose: "数字分身对话", risk: "中"}]`

#### 风险评估标准

| 风险等级 | 标准 | 示例 |
|---------|------|------|
| **高** | 核心功能强依赖、API不稳定、成本高 | Gemini API(对话核心功能) |
| **中** | 重要功能依赖、有降级方案 | Capacitor Camera(头像上传) |
| **低** | 辅助功能依赖、影响范围小 | Three.js(装饰性3D展示) |

### 7. 定义特性详情

为每个特性定义以下内容：

- **特性 ID**：`F-<NNN>`
- **特性名称**：中文名称
- **特性条目类型**：标准特性条目 / 数据类特性条目 / 报表类特性条目（见 7.1）
- **实现要求**：MUST（必须实现）/ SHOULD（建议实现）/ COULD（可选实现）（见 7.2）
- **优先级**：HIGH / MEDIUM / LOW
- **描述**：2-3 句话描述该特性
- **预估复杂度**：SMALL / MEDIUM / LARGE
- **用户角色**：与该特性交互的角色
- **关联系统**：该特性涉及的所有系统列表
- **外部服务依赖**：该特性依赖的外部服务清单（AI/Capacitor/Three.js等）
- **合规类别**：该特性涉及的合规要求（见 7.4）
- **风险等级**：高 / 中 / 低（见 7.5）
- **监管报送**：是否涉及向监管机构报送（见 7.6）
- **可用性要求**：可用性等级（见 7.7）
- **数据敏感等级**：数据敏感程度（见 7.8）
- **认证级别**：身份认证要求（见 7.9）
- **依赖关系**：该特性所依赖的其他特性 ID（如有）

#### 7.1 特性条目类型说明

特性条目类型用于区分特性的业务性质，便于后续归类和设计。

| 类型 | 说明 | 判断标准 | 示例 |
|------|------|----------|------|
| **标准特性条目** | 标准业务功能视角的特性 | 面向用户操作、业务流程的功能 | 用户注册、流程审批、订单支付 |
| **数据类特性条目** | 涉及数据使用、数据加工的特性 | 以数据处理为核心目的 | 数据同步、数据清洗、数据统计加工 |
| **报表类特性条目** | 涉及报表生成的特性 | 以报表产出为核心目的 | 统计报表生成、报表导出、报表订阅 |
| **（可扩充）** | 未来可根据业务需要扩展更多类型 | - | - |

#### 7.2 实现要求说明

| 要求 | 说明 | 判断标准 |
|------|------|----------|
| **MUST** | 必须实现，不实现则需求不完整 | MVP 必需、核心业务链路 |
| **SHOULD** | 建议实现，提升产品价值 | 重要增强、用户体验优化 |
| **COULD** | 可选实现，非核心需求 | 增强体验、非关键路径 |

#### 7.3 关联系统记录示例

```
关联系统: 需求管理系统（A+类）、工作流引擎（A类）、通知服务（A类）
```

#### 7.4 合规类别说明

| 类别 | 说明 | 适用场景 |
|------|------|----------|
| **个人信息保护** | 涉及个人信息的收集、存储、使用 | 用户注册、账户管理、交易记录 |
| **反洗钱(AML)** | 大额交易报告、可疑交易监测 | 转账汇款、现金存取、跨境支付 |
| **征信合规** | 涉及征信数据查询和使用 | 贷款审批、额度评估、信用评分 |
| **等保三级** | 网络安全等级保护三级要求 | 核心业务系统、交易系统 |
| **反欺诈** | 交易风控、欺诈行为识别 | 支付交易、账户变更、异常登录 |
| **数据跨境** | 涉及数据出境 | 跨境支付、国际汇款 |
| **无特殊合规** | 不涉及上述合规要求 | 内部管理类功能 |

#### 7.5 风险等级说明

| 等级 | 说明 | 判断标准 |
|------|------|----------|
| **高** | 涉及资金损失、客户信息泄露或监管处罚风险 | 资金交易、密码修改、敏感信息操作 |
| **中** | 可能影响业务连续性或用户体验 | 流程审批、数据查询、状态变更 |
| **低** | 影响有限，不涉及核心业务 | 基础查询、配置管理、辅助功能 |

#### 7.6 监管报送说明

| 类别 | 说明 | 报送场景示例 |
|------|------|-------------|
| **大额交易报送** | 单笔或累计金额超过阈值时向反洗钱系统报送 | 大额转账、现金存取 |
| **可疑交易报送** | 符合可疑交易特征时自动上报 | 频繁小额交易、异常时间交易 |
| **征信查询报送** | 每次征信查询记录报送征信系统 | 贷款审批、额度审核 |
| **税务报送** | 涉税交易信息向税务系统报送 | 代扣代缴、税务申报 |
| **无报送** | 不涉及监管报送要求 | 内部管理、辅助功能 |

#### 7.7 可用性要求说明

| 等级 | 可用性指标 | 适用场景 |
|------|-----------|----------|
| **99.99%** | 年停机时间 < 52 分钟 | 核心交易系统（支付、清算） |
| **99.95%** | 年停机时间 < 4.4 小时 | 重要业务系统（开户、贷款） |
| **99.9%** | 年停机时间 < 8.8 小时 | 一般业务系统（查询、统计） |
| **99.5%** | 年停机时间 < 1.8 天 | 内部管理系统 |

#### 7.8 数据敏感等级说明

| 等级 | 说明 | 数据类型示例 |
|------|------|-------------|
| **极高** | 核心机密，未经授权泄露会造成重大损失 | 客户密码、密钥、账户余额 |
| **高** | 敏感数据，需严格访问控制 | 身份证号、手机号、交易记录 |
| **中** | 内部使用数据，限制公开范围 | 操作日志、审批记录、统计数据 |
| **低** | 公开信息，无敏感要求 | 公告信息、公开报表 |

#### 7.9 认证级别说明

| 级别 | 说明 | 适用场景 |
|------|------|----------|
| **L0 强认证** | 多种认证因子组合（密码+U盾+人脸） | 大额转账、密码重置、账户变更 |
| **L1 高认证** | 双因子认证（密码+短信/人脸） | 登录、重要操作确认 |
| **L2 中认证** | 单因子增强认证（密码+设备绑定） | 一般操作、设备管理 |
| **L3 基础认证** | 基础密码认证 | 查询、浏览等低风险操作 |

### 8. 分析特性依赖关系

识别并记录特性间的依赖关系：

- **顺序依赖**：特性 B 需要在特性 A 完成后才能开始
- **数据依赖**：特性 B 依赖特性 A 创建的数据模型
- **集成依赖**：特性 B 需要特性 A 提供的 API
- **无依赖**：该特性可独立开发

### 9. 保存特性列表

将特性列表保存至：
- 路径：`.asdm/workspace/requirements/REQ-<id>-<name>/feature-list.md`
- 格式：遵循 `.asdm/toolsets/req-analyzer/spec/feature-list-spec.md` 中的模板

### 10. 更新需求列表

更新需求汇总列表：
- 路径：`.asdm/workspace/requirements/requirements-list.md`
- 将需求状态从 "CLARIFIED" 更新为 "BREAKDOWN"

## 执行指南

### 何时使用本操作

在以下情况下使用本操作：
- 业务需求的澄清已完成
- 需要将业务需求分解为可实现的特性
- 希望对特性进行优先级排序和估算以用于规划
- 为原型生成或文档编制准备输入

### 拆解质量指南

#### 核心原则（必须遵循）

1. **业务价值导向**
   - 每个特性都必须能回答"上线后用户能获得什么价值"
   - 避免技术实现视角的拆解

2. **独立可交付**
   - 特性可以独立开发、测试、部署
   - 上线后能产生完整的业务价值
   - 不依赖其他特性的完成

3. **合理的粒度**
   - 不能太细：避免变成开发任务或技术组件
   - 不能太粗：避免包含多个独立的业务价值点
   - 目标：3-8个特性,每个都是完整的业务场景

4. **用户视角命名**
   - 特性名称体现"用户可以做什么"
   - 避免技术术语和实现细节

#### 质量检查清单

在完成拆解后,使用以下清单自检：

- [ ] 每个特性都能独立上线并产生业务价值
- [ ] 每个特性都可以用"用户可以XXX"完整描述
- [ ] 没有技术组件作为独立特性（如"数据库"、"接口"、"页面"）
- [ ] 没有开发任务作为独立特性（如"实现算法"、"开发接口"）
- [ ] 特性数量在3-8个范围内
- [ ] 特性之间有清晰的边界,没有重叠
- [ ] 所有澄清的需求都被特性覆盖
- [ ] 每个特性都有明确的用户角色和使用场景
- [ ] 特性的优先级反映了业务价值的高低

### 特性数量指南

- **理想范围**：3-8个特性
- **可接受范围**：2-10个特性（需评估粒度是否合适）
- **超过10个**：需要重新调整拆分粒度或拆分需求
- **低于2个**：检查是否过粗,可能需要进一步拆分

### 拆解示例对比

#### ❌ 错误示例（过细,技术视角）

**需求**：审批历史意见引用

**错误的特性拆解**：
- F-001：历史意见查询接口
- F-002：历史意见展示界面
- F-003：关键字提取与匹配
- F-004：历史意见引用审计

**问题**：
- 这些都是技术组件,不是完整的业务功能
- 用户看不到"接口"、"界面"、"算法",只能看到"历史意见引用"这个完整功能
- 无法独立上线,必须一起交付

#### ✅ 正确示例（合适,业务视角）

**需求**：审批历史意见引用

**正确的特性拆解**：
- F-001：审批历史意见引用

**说明**：
- 这是一个完整的业务功能,包含从用户点击按钮到看到历史意见的完整体验
- 上线后用户能立即获得价值
- 内部的技术实现（接口、界面、算法）都在这个特性范围内

**如果确实需要拆分**（例如分阶段交付）：
- F-001：审批历史意见查询与展示（MVP版本,基础查询功能）
- F-002：关键字智能匹配（增强版本,提升查询准确性）

**说明**：
- 每个特性都是完整的功能,可以独立上线
- F-001 上线后用户就能用基础功能
- F-002 是增强功能,上线后能提升用户体验

## 使用方式

使用本指令时，AI 模型应：
1. 检测响应语言
2. 从用户处获取需求 ID
3. 校验澄清说明是否存在
4. **检测增量更新场景**（检查 feature-list.md 是否已存在）
5. 加载并分析澄清说明
6. 读取项目上下文（如有）
7. 识别特性
8. 校验特性数量
9. 识别关联系统
10. 定义特性详情（含关联系统）
11. 分析特性依赖关系
12. 保存特性列表
13. 更新需求列表
14. 向用户展示摘要
