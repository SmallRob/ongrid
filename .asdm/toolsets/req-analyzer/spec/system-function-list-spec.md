# 系统功能清单规范

## 语言指南

本系统功能清单必须使用环境检测到的响应语言。请确保本文档中的所有内容遵循以下规则：

1. **语言一致性**：整个文档使用同一种语言
2. **写作规范**：遵循所选语言的写作风格和格式规范
3. **清晰表达**：确保内容清晰易懂

**支持的语言**：
- 中文（zh）
- 英文（en）
- 其他基于环境检测的语言

---

## 概述

本规范定义了按关联系统聚合的功能清单的结构和格式要求。系统功能清单用于将 `req-breakdown` 阶段产出的特性列表，按关联系统进行聚合分析，明确各系统的职责边界、接口依赖和集成关系。

**主要用途**：
- 明确各系统的功能归属（自研/集成/委托/待定）
- 定义系统间的接口规范和数据交换格式
- 支持按系统划分开发团队的工作范围
- 为集成测试和联调提供依据

## 6. 数据交换规范

### 6.1 数据实体清单

| 数据实体 | 涉及系统 | 交换方向 | 同步方式 | 频率 | 说明 |
|----------|----------|----------|----------|------|------|
| 用户信息 | 本系统 ↔ 用户中心 | 双向 | 同步 | 实时 | 用户注册时同步 |
| 医生信息 | 本系统 ↔ HIS | 接收 | 增量同步 | 每日 | 从HIS同步医生数据 |
| 支付订单 | 本系统 ↔ 支付系统 | 双向 | 同步+异步 | 实时 | 支付结果异步回调 |
| 预约记录 | 本系统 ↔ HIS | 发送 | 同步 | 实时 | 预约成功后同步 |
| 短信记录 | 本系统 → 短信平台 | 单向 | 同步 | 实时 | 发送通知短信 |

### 6.2 数据同步策略

#### 6.2.1 实时同步
适用于：需要即时获取最新数据的场景
- 用户信息同步
- 支付结果回调

#### 6.2.2 定时同步
适用于：允许一定延迟的数据同步
- 医生排班同步
- 统计数据汇总

#### 6.2.3 增量同步
适用于：数据量大但变化较少的场景
- 用户基础信息
- 配置数据

### 6.3 数据格式规范

所有接口数据格式遵循以下规范：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "timestamp": 1609459200000,
  "requestId": "uuid-string"
}
```

## 章节指南

### 1. 系统总览指南

提供所有涉及系统的高层概览。

#### 指导原则

- 表格应包含每个系统的基本信息
- 系统按类型分组（本系统 → 内部 → 外部 → 基础设施）
- 统计信息帮助快速了解整体工作量

#### 重要事项

- 系统ID 格式：`SYS-001`, `SYS-002`...
- 系统类型必须准确区分
- 归属团队应明确到具体负责人

### 2. 功能映射总表指南

展示 Feature 到系统的映射关系。

#### 指导原则

- 每个功能要点都应映射到具体系统
- 归属类型必须准确（自研/集成/委托/待定）
- 优先级应与 Feature 保持一致

#### 映射规则

| 功能类型 | 归属判断 |
|----------|----------|
| 本系统用户操作 | 自研 |
| 调用第三方API | 集成 |
| 数据推送到其他系统 | 集成 |
| 接收其他系统数据 | 集成 |
| 委托其他团队开发 | 委托 |
| 无法确定归属 | 待定 |

### 3. 本系统功能详情指南

详细描述本系统的功能范围和模块划分。

#### 指导原则

- 明确本系统的职责边界
- 功能划分应符合高内聚低耦合
- 开发顺序应基于依赖关系

### 4. 外部系统功能详情指南

描述每个外部系统的集成需求。

#### 指导原则

- 提供足够的上下文帮助理解外部系统
- 接口定义应具体可执行
- 明确双方的责任边界

### 5. 接口规范指南

定义系统间的接口标准。

#### 指导原则

- 接口定义遵循 RESTful 规范
- 请求/响应格式统一
- 错误码定义清晰

#### 接口ID 命名规则

- 出接口：`API-XXX-OUT`（如 API-001-OUT）
- 入接口：`API-XXX-IN`（如 API-002-IN）

## 使用指南

当创建系统功能清单时：

1. **识别系统**：从特性列表中识别所有涉及的关联系统
2. **分类整理**：按系统类型分类（本系统/内部/外部/基础设施）
3. **功能映射**：将每个功能要点映射到具体系统
4. **定义接口**：识别并定义需要对接的接口
5. **明确责任**：确定每个功能的归属团队
6. **制定计划**：基于依赖关系制定开发顺序

**重要事项**：
- 系统边界必须清晰，避免模糊地带
- 接口定义应具体，包含请求/响应示例
- 对于不确定的事项，标记为「待确认」并列入跟踪

## 最佳实践

使用此规范时：

1. **系统识别完整**：确保不遗漏任何涉及的关联系统
2. **归属判断准确**：每个功能都有明确的归属类型
3. **接口定义具体**：包含完整的请求/响应示例
4. **责任边界清晰**：明确本团队和其他团队的工作范围
5. **风险提前识别**：尽早发现集成风险并制定应对措施

### 常见陷阱

- **陷阱 1**：遗漏某些关联系统 → 应该逐个 Feature 分析涉及的系统
- **陷阱 2**：接口定义过于模糊 → 应该包含具体的字段定义和示例
- **陷阱 3**：责任边界不清 → 应该明确到具体团队和联系人
- **陷阱 4**：忽视数据同步的时序问题 → 应该明确同步策略和频率
