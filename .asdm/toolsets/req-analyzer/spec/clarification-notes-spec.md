# 澄清记录规范

## 语言指南

本澄清记录必须使用环境检测到的响应语言。请确保本文档中的所有内容遵循以下规则：

1. **语言一致性**：整个文档使用同一种语言
2. **写作规范**：遵循所选语言的写作风格和格式规范
3. **清晰表达**：确保内容清晰易懂

**支持的语言**：
- 中文（zh）
- 英文（en）
- 其他基于环境检测的语言

---

## 概述

本规范定义了需求澄清记录的结构和格式要求。澄清记录用于记录对业务诉求的结构化问题分析，包括按维度分类的澄清问题、假设条件列表、待确认事项，为后续的特性拆解提供清晰的基础。

**主要用途**：
- 系统化地消除需求歧义
- 记录关键假设和约束条件
- 标记需要业务方确认的事项
- 为特性拆解阶段提供完整的上下文

## 章节指南

### 2. 业务目标维度指南

用于澄清需求的商业驱动力、目标和期望成果。

#### 指导原则

- 聚焦商业价值而非具体实现方式
- 确保理解清楚预期的业务成果和 KPI
- 识别与现有业务的关系和影响

#### 典型问题示例

- 这个需求的核心商业目标是什么？
- 预期的业务成果或 KPI 是什么？
- 项目的时间紧迫性如何（是否有固定截止日期）？
- 这个需求如何与现有的业务目标对齐？

#### 例子

```markdown
## 2. 业务目标维度

### 2.2 澄清问题列表

| 问题编号 | 问题 | 预期回复 | 影响范围 | 优先级 |
|----------|------|----------|----------|--------|
| BG-001 | 打造在线挂号系统的首要商业目标是什么？是为了提高患者转化率、降低线下成本，还是提升医院品牌？ | 明确的商业目标描述 | 整体系统设计 | HIGH |
| BG-002 | 预期在系统上线后的第一年，挂号量达到什么规模？ | 具体的年度或月度目标数字 | 容量规划 | HIGH |
```

### 3. 用户角色维度指南

用于澄清不同用户群体及其需求和交互模式。

#### 指导原则

- 识别所有关键用户群体
- 理解每个角色的目标和操作流程
- 确保不同角色的需求被准确理解

#### 典型问题示例

- 系统的主要用户角色有哪些？
- 不同角色的权限和操作能力有什么区别？
- 每个角色的主要使用场景和工作流是什么？
- 是否存在不同用户对同一功能的不同期望？

#### 例子

```markdown
## 3. 用户角色维度

### 3.1 用户角色列表

| 角色名称 | 角色描述 | 特征 | 使用场景 |
|----------|----------|------|----------|
| 患者 | 需要预约挂号的患者 | 大多数是移动端用户，追求简单易用 | 选择医生、时间段预约 |
| 医生 | 需要管理自己的排班和患者信息的医生 | 需要高效管理工具 | 查看日程、应诊 |
```

### 4. 功能边界维度指南

用于澄清功能范围、优先级和集成需求。

#### 指导原则

- 明确说明哪些功能必须做，哪些明确不做
- 理解与其他系统的集成点
- 确定用户交互流程的深度

#### 典型问题示例

- 这个需求涵盖的核心功能有哪些？什么是明确不包括的？
- 系统是否需要与现有的医院信息系统（HIS）集成？
- 支付流程是由我们实现还是接入第三方？
- 用户查看预约历史、取消预约、改期等操作是否在范围内？

#### 例子

```markdown
## 4. 功能边界维度

### 4.1 核心功能范围

**必须包含**：
- 患者预约挂号流程
- 医生排班管理
- 患者查看预约信息

**明确排除（超出范围）**：
- 医疗诊断功能
- 患者病历系统
```

### 5. 非功能约束维度指南

用于澄清性能、安全、可用性等技术和运营约束。

#### 指导原则

- 理解系统的性能、安全和可靠性需求
- 识别任何技术栈的限制
- 确保对 SLA 和可用性期望的共识

#### 典型问题示例

- 系统的日活用户预期是多少，峰值并发访问量是多少？
- 对数据安全和患者隐私有什么特殊要求（医疗监管）？
- 系统的可用性要求是什么（99.9% 还是 99.99%）？
- 是否有特定的技术栈限制（如必须支持的浏览器版本）？

#### 例子

```markdown
## 5. 非功能约束维度

### 5.1 约束列表

| 约束类型 | 具体约束 | 优先级 |
|----------|----------|--------|
| 性能 | 预约提交应在 2 秒内响应 | HIGH |
| 安全 | 患者隐私信息必须加密存储 | HIGH |
| 可用性 | 系统可用性目标 99.9% | MEDIUM |
```

## 使用指南

当记录需求澄清时：

1. **逐维度收集**：按四个维度逐一列出澄清问题，确保覆盖完整
2. **获取业务方确认**：为每个问题收集业务方的明确回复
3. **标记确认状态**：确认已解答的问题，标记仍需确认的待办项
4. **记录假设**：针对无法立即确认的假设，标注置信度和风险
5. **定期更新**：澄清记录应随着业务方的反馈不断更新

**重要事项**：
- 问题应具体且可回答，避免模糊表述
- 优先级应基于对功能设计的影响程度
- 待确认事项应有明确的所有者和截止日期
- 假设应明确记录其置信度和风险影响

## 最佳实践

使用此规范时：

1. **问题质量**：确保每个问题都能得到清晰的回答，避免是非题
2. **维度平衡**：不同维度的问题数量应相对平衡（避免某个维度过多问题）
3. **优先级排序**：高优先级问题应该首先获得确认
4. **及时跟进**：对待确认项应定期跟进，避免需求卡住
5. **信息完整**：确保每条信息都有足够的上下文支持后续工作

### 常见陷阱

- **陷阱 1**：提出过于宽泛的问题 → 应该分解为具体的、可回答的问题
- **陷阱 2**：忽视非功能约束 → 应该主动澄清性能、安全等约束
- **陷阱 3**：不跟进待确认事项 → 应该设置提醒，定期检查确认状态
- **陷阱 4**：澄清记录过时不更新 → 应该随业务反馈持续更新
