# 需求说明书规范（业技数智融合标准版）

## 语言指南

本需求说明书必须使用环境检测到的响应语言。请确保本文档中的所有内容遵循以下规则：

1. **语言一致性**：整个文档使用同一种语言
2. **写作规范**：遵循所选语言的写作风格和格式规范
3. **清晰表达**：确保内容清晰易懂

**支持的语言**：
- 中文（zh）
- 英文（en）
- 其他基于环境检测的语言

---

## 概述

本规范基于业技数智融合需求说明书模板，定义了需求文档的结构和格式要求。需求说明书由 3 个一级章节、12 个二级章节组成。

**模板使用说明**：
- 编写过程由粗到细，逐步完善
- 非标准类特性条目如不涉及可直接删除
- 其他章节中没有涉及的内容，可以直接填写"不涉及"，但不可删除目录
- 可根据实际情况补充其他需要说明的内容
- **性能容量、业务连续性、兼容性、可移植性、易用性等非功能需求、安全需求以及数据输出活动评估、合规要求均已实现线上化，需求文档中无需填写，均以线上内容为准**
- 版本历史为必填项，确保文档可追溯性
- 非新系统建设需求，参考文档中原需求文档名称和查阅路径为必填

**主要用途**：
- 为业务方和产品团队提供最终需求确认文件
- 为开发团队提供完整的需求和验收标准
- 归档需求分析全过程
- 支持跨部门沟通和协作

## 章节指南

### 1. 概述指南

#### 1.1. 背景、现状和业务目标

提供文档的基本信息和上下文。

**指导原则**：
- 背景应简洁但充分，解释"为什么需要"而非"怎么做"
- 业务目标应具体、可衡量
- 应从业务视角出发描述问题和价值

**重要事项**：
- 背景应基于真实的业务痛点
- 目标应与澄清阶段的讨论一致

**示例**：
```markdown
### 1.1. 背景、现状和业务目标

加强零售战略闭环管理：落地零售战略管理体系的建设，更好的支持零售战略闭环管理，实现零售战略闭环管理的流程化、标准化及系统化机制。

资产沉淀及线上化业务流程：零售业务板块在战略规划、专题方案等文档的统筹管理上，如分类、存储及查阅等环节，缺乏必要的系统支持。

总体建设目标：以"战略检视"为主要目标，沿零售战略闭环管理流程，从规划、解析等环节中沉淀资产，为战略检视提供必要输入，形成检视结论，支撑战略迭代，做实闭环管理。
```

### 2. 业务需求总述指南

清晰表述系统的整体需求和功能分布。

**指导原则**：
- 整体业务流程应覆盖主要场景
- 功能层级表应从大类逐级分解
- 特性条目清单应明确优先级
- 关联系统应明确配合改造点和测试责任

### 3. 特性条目详细说明指南

详细定义系统应实现的功能。

**指导原则**：
- 每个特性应有清晰的业务规则表
- 应包含原型输入输出要素表
- 应说明操作角色及权限
- 应列出异常场景和处理方式

**重要事项**：
- 非标准类特性条目如不涉及可直接删除
- 其他章节中没有涉及的内容，填写"不涉及"但不可删除目录
- 可根据实际情况补充其他需要说明的内容

## 最佳实践

使用此规范时：

1. **结构清晰**：使用明确的层级结构便于导航
2. **表格清晰**：表格应包含必要信息但不过度详细
3. **用户视角**：描述应从用户需求而非技术实现出发
4. **及时更新**：文档应随需求确认不断更新
5. **版本可追溯**：版本历史为必填项

### 常见陷阱

- **陷阱 1**：特性类型未区分 → 必须明确标注标准/数据类/报表类，使用对应模板
- **陷阱 2**：业务规则缺失 → 每个标准特性必须包含结构化的业务规则表
- **陷阱 3**：关联系统描述不清 → 必须明确系统标识、配合测试责任和改造点
- **陷阱 4**：异常处理遗漏 → 每个特性必须列出异常场景，无异常时明确标注"不涉及"
- **陷阱 5**：版本历史未填写 → 版本历史为必填项，确保可追溯性
- **陷阱 6**：删除了不涉及的章节目录 → 应填写"不涉及"，但不可删除目录
